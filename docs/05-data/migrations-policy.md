# 마이그레이션 정책

> 데이터베이스 스키마 변경 시 반드시 따라야 할 규칙.

## 도구
- ORM 마이그레이션 도구 사용 (Prisma Migrate / Alembic / Flyway 등)
- 모든 마이그레이션은 코드로 관리 (수동 SQL 금지)

## 명명 규칙
```
YYYYMMDD_HHMMSS_description.sql
20260415_103000_add_users_email_index.sql
```

## Backward Compatibility 원칙

### 항상 안전한 변경
- 새 테이블 추가
- 새 컬럼 추가 (NULL 허용 또는 기본값)
- 인덱스 추가 (CONCURRENTLY 옵션)
- 새 enum 값 추가

### 조심해야 할 변경
- NOT NULL 추가 → 두 단계로: ① 기본값 + 백필 → ② NOT NULL 적용
- 컬럼 타입 변경 → 새 컬럼 추가 + 백필 + 코드 전환 + 구 컬럼 제거

### 위험한 변경 (Expand-Migrate-Contract 패턴)
컬럼/테이블 이름 변경, 삭제는 절대 한 번에 하지 말 것.

```
1. Expand: 새 컬럼/테이블 추가 (구 + 신 공존)
2. Migrate: 코드를 신 스키마 사용하도록 점진 전환
3. 검증: 모든 트래픽이 신 스키마 사용 확인 (1주~1달)
4. Contract: 구 스키마 제거
```

## Zero-Downtime 마이그레이션 체크리스트
- [ ] 마이그레이션이 5초 이내 완료되는가?
- [ ] 5초 초과 시 청크 단위 백필 사용했는가?
- [ ] 락이 발생하는가? (CONCURRENTLY, batch 크기 조정)
- [ ] 롤백 시나리오 검증했는가?
- [ ] 스테이징에서 프로덕션과 유사한 데이터로 테스트했는가?

## 금지 사항
- ❌ 마이그레이션 파일 수정 (한 번 머지된 것은 불변)
- ❌ 마이그레이션 안에서 서비스 코드 호출
- ❌ 트랜잭션 없이 여러 DDL 묶기
- ❌ 프로덕션에서 직접 SQL 실행 (긴급 시 별도 승인)

## 데이터 백필
대량 백필은 별도 작업으로 분리:
1. 스키마 변경 (구조만)
2. 백그라운드 잡으로 백필
3. 검증
4. 코드 전환

## 롤백 정책
- 마이그레이션 직후 30분 모니터링
- 문제 발견 시 **forward fix 우선**, rollback은 최후 수단
- Down 마이그레이션은 작성하되, 실제 사용은 신중히
